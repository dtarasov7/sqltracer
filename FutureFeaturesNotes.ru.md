# Заметки по спорным будущим функциям

Ниже сохранены рабочие рассуждения по четырем спорным функциям, которые пока не очевидно нужны в `sqltracer`.

## 1. Daemon + Separate Client

Что это значит:

- `daemon` живет рядом с приложением или БД и собирает трафик
- `client` подключается к нему отдельно и только отображает данные

Польза:

- TUI не зависит от жизненного цикла прокси
- можно подключаться к удаленной машине без `ssh` в интерактивный экран
- падение UI не роняет сбор трафика
- можно запускать один daemon и несколько клиентов
- проще хранить историю дольше, чем живет текущий TUI-сеанс

Когда это реально нужно:

- удаленные серверы
- shared env для команды
- long-running capture
- хочется отделить `сбор` от `просмотра`

Минусы:

- архитектура заметно усложняется
- нужен протокол между `daemon` и `client`
- появляется вопрос auth, reconnection, version compatibility
- для локального single-user сценария это часто оverkill

Вывод для `sqltracer`:

- сейчас это не `must-have`
- становится нужным только если нужно смотреть трафик не на той же машине, где крутится TUI, или держать постоянный capture отдельно от UI

## 2. Web UI

Что это дает:

- просмотр в браузере
- удобнее для людей, которые не любят TUI
- проще делиться ссылкой внутри команды
- легче делать richer inspector, табы, фильтры, графики, длинные explain/result panes

Практическая польза:

- onboarding проще
- для analytics и timeline браузер действительно удобнее
- длинные ответы, таблицы и поиск по ним делать проще, чем в `curses`

Минусы:

- это уже почти отдельный продукт
- нужен backend streaming/API
- нужен frontend state management
- надо думать про auth, CSRF, network exposure
- много работы будет не в анализе SQL, а в web-обвязке

Когда это окупается:

- если инструментом пользуются несколько человек
- если нужна демонстрация и шаринг
- если TUI начинает ограничивать по UX

Вывод:

- полезно, но не сейчас
- для текущего этапа это скорее дорогой второй интерфейс к уже не до конца зрелому ядру

## 3. gRPC / Remote API

Это не UI-фича, а фундамент.

Что дает:

- внешний интерфейс к событиям
- можно подключать не только TUI, но и Web UI, exporter, alerter, тестовый клиент
- легче строить `daemon/client` архитектуру
- можно интегрироваться с внешними системами

Практическая польза:

- API для стриминга событий
- API для explain/export/filter на стороне клиента
- единый контракт между сборщиком и отображением

Почему это бывает важно:

- как только появляется больше одного consumer данных, без API начинается каша
- иначе логика размазывается между TUI, no-TUI, file export, Web UI и будущими интеграциями

Минусы:

- если есть один процесс и один UI, это преждевременное усложнение
- придется стабилизировать event model
- нужно поддерживать совместимость версий

Вывод:

- само по себе `gRPC` пользователю не нужно
- оно имеет смысл только если принято решение делать `daemon/client` или `Web UI`
- до этого момента лучше держать внутренний event model чистым, но не тащить network API

## 4. CI mode

Это самая практичная из четырех функций.

Что это обычно значит:

- запуск без интерактива
- прогон тестового сценария
- сбор SQL-событий
- автоматический fail по правилам:
  - есть `N+1`
  - есть `slow query`
  - есть unexpected full table scan
  - выросло число запросов
  - появился forbidden query pattern

Польза:

- превращает инструмент из ручного дебаггера в контроль качества
- ловит регрессии до продакшена
- особенно полезно для ORM-heavy кода и API integration tests

Когда реально полезно:

- есть автотесты, гоняющие приложение через БД
- есть конкретные правила, которые нужно валить в CI
- команда готова поддерживать baseline и thresholds

Минусы:

- нужны стабильные правила, иначе будет много шума
- без хорошей фильтрации и нормализации запросов `CI mode` быстро раздражает
- нужно продумать формат отчета

Вывод:

- из этих четырех функций именно `CI mode` выглядит самым практичным кандидатом на будущее
- но только после стабилизации event capture и фильтрации

## Общий приоритет

Если смотреть прагматично для `sqltracer`, порядок такой:

1. `CI mode`
2. `Daemon + separate client`
3. `gRPC / remote API`
4. `Web UI`

Почему так:

- `CI mode` дает самую прямую прикладную ценность
- `daemon/client` нужен, если инструмент становится не только локальным
- `gRPC` нужен как следствие `daemon/web`, а не как самоцель
- `Web UI` самый дорогой по трудозатратам и не решает проблемы ядра

## Короткий итог

Если нет уверенности, что эти функции действительно нужны, сейчас не стоит спешить с:

- `Web UI`
- `gRPC`
- `daemon/client`

Первым кандидатом на будущее имеет смысл рассматривать именно `CI mode`, и то после стабилизации текущего PostgreSQL ядра и UX.
